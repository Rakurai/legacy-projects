# What the generated_docs Fields Unlock for the Migration Server

> **Context:** The `generated_docs/` files are the output of the doc_gen pipeline's
> four-pass LLM annotation process. This document explores what additional value the
> less-obvious fields — `notes`, `rationale`, `usages`, and to a lesser extent the
> `details` field in combination — can provide at MCP server **build time** to support
> views that the existing V1 entity model doesn't yet expose.
>
> The framing throughout is: what can be derived or pre-computed **without additional
> LLM inference**, and what does each derivation enable for planning, spec-creating,
> and auditor agents?

---

## 1. What the Fields Actually Encode

Before reasoning about uses, a precise characterization of what each field contains in
practice (distinct from the formal definitions in `artifacts/artifacts.md`):

**`brief` + `details`** — The implementor's view. What this function does and how it
does it, derived from reading its source code and the briefs of its callees. This is
the forward-pass output: bottom-up reasoning about internals. It answers "if you read
the code, here is what you would conclude."

**`notes`** — Behavioral anomalies and invariants the LLM surfaced while reading the
code. Edge-case behavior, unexpected side effects, preconditions the caller must
satisfy, lifecycle constraints, and design quirks that don't fit the main description.
Notes exist precisely because the LLM noticed something worth flagging that didn't
belong in brief or details.

**`rationale`** — The caller's view. Synthesized during the refinement pass from the
`usages` evidence: "what do callers of this function use it for?" It answers "if you
look at every call site, here is the pattern." This is fundamentally different from
`details` — it reflects the external perspective on the function, not the internal one.

**`usages`** — The raw call-site evidence underlying `rationale`. For each caller, a
natural-language description of exactly what that caller is doing with this entity.
This is the most granular external evidence in the corpus: ~55% of entities have it,
concentrated on the well-connected functions that matter most for migration.

These four fields were generated by distinct passes using distinct evidence. They are
complementary angles on the same entity, not redundant restatements of each other.
The key insight is that **the MCP server currently treats them as opaque text columns**
— they're stored, they're full-text searched, and they're returned in entity records.
But they encode structured semantic views that the build pipeline could materialize as
derived artifacts.

## 2. Notes → Edge Case Catalog (Zero New LLM Work)

### What notes contain

The `notes` field for well-documented functions contains things like:

- Invariants the caller must satisfy: "this function should not be called if the
  character is already in the extraction list"
- Ordering dependencies: "the caller must have computed AC modifiers before calling
  this; the result depends on perm_ac having been set"
- Side effects beyond what the call graph captures: "this function modifies the
  victim's position as a side effect of zero-HP detection"
- Double-edged behaviors: "this variable is reused as both the head of the list and
  a traversal pointer; modifications during iteration require care"
- Global state dependencies that aren't visible as explicit parameters

For 83% of entities, this field is populated. For the ~850 entities in the capability
groups — the functions that actually matter for migration — coverage is essentially
complete, because these are precisely the well-connected functions that generated the
richest LLM outputs during refinement.

### What this enables at build time

**Per-capability edge case aggregation.** At build time, collect `notes` from all
entities assigned to each capability group. The result is a pre-computed edge case
register for each system — not synthesized by an LLM, but drawn from the existing
per-function observations. For a capability like `combat`, the notes from `one_hit`,
`damage`, `raw_kill`, `multi_hit`, `check_dodge`, and their peers collectively encode
the edge case landscape of the combat system.

This matters directly because the consensus (`§4.3`) requires every spec dossier to
include at least 3-5 edge cases per spec. The spec agent writing a "hit resolution"
dossier currently has to discover edge cases by reading source code. If the notes from
all hit-resolution-scope functions are aggregated and surfaced as part of
`get_chunk_evidence`, the agent starts with a pre-populated edge case list rather than
a blank page.

**Notes as a complexity signal.** The presence, length, and density of notes on a
function is a meaningful signal for migration risk. A function with five sentences of
notes has more documented quirks than a function with none. At build time, compute
`notes_complexity` (e.g., character count, or a normalized 0-1 score) as a derived
metric alongside `fan_in`/`fan_out`. The planning agent's `estimate_complexity` tool
can then incorporate this: a system where 60% of functions have substantial notes is
harder to spec faithfully than one where notes are sparse.

**Notes for invariant flagging.** Notes that contain precondition language
("must be", "requires", "should not be called if", "assumes") can be detected with a
simple heuristic at build time. Flagged entities appear in the `explain_interface`
tool response under a "caller responsibilities" heading — seeding the precondition
fields of the Wave 1 infrastructure contracts without requiring additional LLM passes.

### Concrete build-time implementation

Add a column `notes_length INT` to the entities table (build-time aggregation, not
LLM). Add a `capability_edge_cases JSONB` column (or a separate table) to the
capabilities table, populated by: for each capability, collect notes from all member
entities, deduplicate semantically similar ones, and store as a ranked list (longer
notes first as a quality proxy). This is a pure aggregation pass over existing data —
no new model calls.

## 3. Rationale → Pre-Populated Contract Seedlings

### What rationale actually is

The `rationale` field is derived from the `usages` evidence: "given how every caller
uses this function, what is the behavioral contract from the consumer's perspective?"
The refinement pass synthesizes usages into a coherent statement of role.

For infrastructure-level functions — `damage()`, `apply_affect()`, `game_act()` (if
documented), `extract_char()`, `update_pos()` — the rationale is the closest thing in
the corpus to a pre-written API contract clause. For a function like `damage()` with
dozens of callers spread across combat, magic, and affects, the rationale will say
something like "used when any effect that deals HP damage needs to be applied to a
target — covers both spell damage and physical combat damage, and is the authoritative
path for all damage application because it handles position updates, death detection,
and messaging."

That sentence is a behavioral contract in prose form. It says: the Evennia equivalent
of `damage()` must be the single dispatch point for all HP reduction, it must update
position, it must detect death, and it must handle messaging.

### What this enables

**Seeding infrastructure contracts.** The consensus commits to a contract registry
(`migration/contracts/`) for the seven Wave 1 infrastructure pieces. Rationale from the
relevant entities is the most direct pre-existing evidence for what those contracts
must guarantee. Before any agent reads source code, the rationale of `damage()`,
`apply_affect()`, `extract_char()`, `update_pos()`, and the major messaging functions
can populate the `summary` and `behavioral_guarantees` fields of the contract stubs.

This is not the same as writing the contract — that still requires source verification.
But it means the spec agent starts from a partial contract populated with the
caller-derived behavioral expectations, not from a blank YAML template.

**Rationale as the `explain_interface` foundation.** The consensus and all three Round
3+mcp responses converge on an `explain_interface` tool that gives a contract-focused
view of an entity. The rationale field is exactly this: the external, caller-side
behavioral description. At tool response time, `explain_interface` should present
`rationale` under a "contract: when and why callers use this" heading, distinct from
`details` (the internal mechanism view). The separation between these two perspectives
is already baked into the doc structure — it just needs to be presented differently in
the tool response.

**Rationale for spell pattern pre-clustering.** For `spell_*` entry points, rationale
text like "used when a caster needs to apply a direct-damage effect to a single target
in the room" or "used to apply a duration-based debuff that reduces target's combat
effectiveness" are natural cluster labels. At build time, embed rationale text for all
`spell_*` and `spec_*` entities and run a lightweight clustering pass (k-means or
agglomerative, k=6-8). The output pre-seeds the `get_spell_cluster` tool's cluster
assignments, which then get refined by the call-graph-based behavioral analysis.
Rationale-based clusters will converge with call-graph clusters for the
pattern-fitting spells (damage, affect application, heal, transport) and diverge for
bespoke ones — the divergence itself flags candidates for the bespoke list.

**Rationale quality as documentation completeness signal.** An entity whose rationale
says "This function exists to facilitate various operations within the codebase" vs.
one that says "Used exclusively in combat resolution, called by multi_hit() to handle
single attack resolution including hit probability, damage calculation, and outcome
messaging" signals completely different doc quality. At build time, compute a
`rationale_specificity` heuristic (e.g., length + presence of capability-specific
terminology) as a complement to the existing doc quality signals.

### Concrete build-time implementation

Store `rationale` in the entities table as-is (already done). Add it to the tsvector
at weight B alongside `details` (currently it may not be indexed separately — worth
verifying). Generate rationale embeddings as part of the standard entity embedding
pass using `Document.to_doxygen()` (already included). Add a build-time pass that:
(1) flags all entities with `fan_in > 5` AND non-null rationale as "contract
seedlings" — store in a `is_contract_seed BOOLEAN` column; (2) for `spell_*`/`spec_*`
entities with non-null rationale, embed rationale text separately into a
`spell_rationale_embedding vector(768)` column for the clustering pass.

## 4. Usages → Semantic Integration Surface

### What usages encode

The `usages` dict maps `"{compound_id}, {caller_signature}"` to a sentence describing
how that specific caller uses the entity. For a function like `apply_affect()` with
`usages` coverage across dozens of callers spanning combat, magic, and skill
functions, this is a pre-computed inventory of every integration point.

The critical property: usages encode **cross-capability calling semantics**. When a
`combat` function appears in the usages of an `affects` function, the description says
not just "combat calls affects" (which the edge count already tells us) but "combat
uses apply_affect() to temporarily reduce a target's armor class after a successful
shield bash." This semantic dimension is what the integration surface tools need.

### What this enables

**Semantic integration surface.** The planned `get_integration_surface` tool answers
"what does system X expose to system Y and how?" Currently this would be answered by
counting `CALLS` edges across capability boundaries. With the usages dict, we can
instead answer it with: for all callee entities in capability Y that have usages from
entities in capability X, aggregate those usage descriptions by theme. The result is
"system X uses system Y to: (1) apply duration-based stat penalties after combat hits,
(2) check active defensive buffs before computing hit probability, (3) trigger damage-
reactive affects on victim death." That is an integration surface description, not
just an edge count.

**Inverting the usages dict for caller analysis.** The usages dict is stored on the
callee — it says "here are my callers and how they use me." At build time, we can
invert this: for each caller, collect all the (callee, usage_description) pairs where
this caller appears in a callee's usages. This inverted view answers "what does this
function use, and specifically why?" — the `explore_entity` tool's dependency context
would be far richer with "damage() calls update_pos() to resolve the character's
position state after HP reduction" vs. just "damage() calls update_pos()."

**Usage pattern deduplication → common calling patterns.** For high-fan-in functions
(damage, apply_affect, extract_char), many callers use them in nearly identical ways.
At build time, cluster usage descriptions for the same callee to identify the
dominant calling patterns. For `damage()`, the patterns might be: direct-damage spell
dispatch, weapon-hit damage application, self-damage effects. These patterns are
directly useful in the spec agent's "extract data contracts" step — the dominant
pattern tells the implementor what the common case must handle, and the minority
patterns flag the edge cases.

**Auditor cross-reference support.** When an auditor checks whether a spec's stated
postconditions match the legacy evidence, the usages dict provides the caller-side
evidence. "Does the spec say that apply_affect() ensures the affect is immediately
visible in subsequent stat queries? Let's check whether the callers assume this." The
usages on apply_affect() from combat functions will say things like "called before
computing hit probability, which reads the affected stat" — confirming the
postcondition. This is evidence the auditor currently has no structured way to access.

### Concrete build-time implementation

Create a secondary `entity_usages` table:

```sql
CREATE TABLE entity_usages (
    callee_id    TEXT REFERENCES entities(entity_id),
    caller_compound TEXT,
    caller_sig   TEXT,
    description  TEXT,
    embedding    vector(768),       -- embed the description for semantic search
    PRIMARY KEY (callee_id, caller_compound, caller_sig)
);

CREATE INDEX idx_usages_callee ON entity_usages(callee_id);
-- For the inverted view (caller→callees):
CREATE INDEX idx_usages_caller_sig ON entity_usages(caller_sig);
```

This materializes the usages dict as a queryable relation rather than an opaque JSONB
blob. It enables:
- `SELECT description FROM entity_usages WHERE callee_id = ?` (how is X used?)
- `SELECT callee_id, description FROM entity_usages WHERE caller_sig LIKE ?` (what does caller Y use and how?)
- Semantic search over descriptions: embed a query like "apply a debuff before combat
  resolution" and find the callee entities whose usage descriptions match

The embedding column enables the `get_integration_surface` and `get_output_templates`
tools to do semantic aggregation rather than just exact lookup.

## 5. Combined Views: The Near-Complete Function Contract

The most powerful insight is that the four doc fields together constitute a near-
complete behavioral contract for well-documented functions — one that the MCP server
currently discards by treating each as an independent text column.

The structure is:

| Contract Component | Source Field | View |
|--------------------|--------------|------|
| What it does | `brief` + `details` | Implementor's view |
| What callers use it for | `rationale` | Consumer's view |
| Specific calling contexts | `usages` (inverted) | Evidence layer |
| What to watch for | `notes` | Risk/edge-case layer |
| Signature + params | `params` + `returns` | Interface layer |

For infrastructure-level functions (the Wave 1 contracts), this five-part structure is
the skeleton of the contract YAML file. The spec agent doesn't derive it from scratch —
they verify and formalize what the doc corpus already asserts.

### The `explain_interface` response shape

The `explain_interface` tool (identified as a high-priority composite tool in the
Round 3+mcp discussions) should present these as distinct sections:

```
explain_interface(entity_id="fn:damage_abc"):
  signature: "void damage(Character *ch, Character *victim, int dam, ...)"

  [from brief+details] What it does:
    Applies dam points of damage to victim, resolves defensive reductions (sanctuary,
    resistances), updates victim's position based on resulting HP, handles death and
    corpse creation, distributes XP, and generates combat messaging.

  [from rationale] Contract: when callers use this:
    The single dispatch point for all HP-reducing effects in the codebase — both
    weapon damage from multi_hit() and spell damage from magic functions route through
    damage(). Callers invoke it as the authoritative path that ensures all downstream
    effects (death detection, position update, XP distribution, messaging) fire in
    the correct order.

  [from notes] Preconditions and invariants:
    - victim must not be in an extract state when called (risk of double-extraction)
    - Sanctuary reduces by half before other modifiers; don't apply sanctuary manually
      before calling this
    - dam=0 is valid; will still trigger miss messaging but no HP change

  [from usages, top patterns]
    - Spell functions: after save-or-take-damage resolution, pass the rolled damage
    - Combat loop: called by one_hit() after hit roll succeeds and raw damage is computed
    - Area effects: called per-target in room iterations for fireball/chain-lightning
```

This response shape is entirely pre-computed from existing fields. It requires no new
LLM pass at build time or query time.

## 6. Details → Searchable Formula Context

The `details` field for computationally dense functions (stat computation, saving
throws, damage formulas, skill checks) contains the LLM's prose interpretation of the
algorithm. This is less precise than reading source code, but it has two properties
that make it useful as a pre-filter:

**Formula-adjacent text is semantically searchable.** "The base damage is computed as
a random roll between minimum and maximum weapon damage, modified by the attacker's
damroll stat and the weapon's skill proficiency" will match a query like "where is
damroll applied to weapon damage" in semantic search, whereas the source code
`dice(obj->value[1], obj->value[2]) + GET_DAMROLL(ch)` will not. The details field
bridges the gap between how agents query (natural language) and how the code is
written (C idioms).

**Details as formula-hunting pre-filter.** For the `get_numeric_constants` and
`get_formula` tools proposed in the discussion, the workflow would be:
1. Semantic search over `details` to find the N functions most likely to implement
   the formula of interest
2. Retrieve source code for those N functions and extract the exact constants

The details field is already indexed in tsvector at weight B. For the formula-hunting
use case, a separate semantic index on `details` alone (excluding brief, notes,
rationale) may give better precision for formula queries than the combined embedding
that mixes all fields. Consider generating a `details_embedding vector(768)` column
distinct from the main `embedding` (which uses `Document.to_doxygen()` combining all
fields). This is one additional build-time embedding pass.

## 7. State as a Trust Signal

The `state` field (`extracted_summary`, `generated_summary`, `refined_summary`,
`refined_usage`) is already in the generated_docs but is not surfaced to consuming
agents. It maps directly to the grounding/trust signals that the discussion rounds
identified as necessary for agents to calibrate trust.

Currently the MCP server dropped `doc_state` from the entity model (spec 005). But
for the migration server's verification needs, the distinction matters:

| State | Trust Level | Agent Implication |
|-------|-------------|-------------------|
| `refined_summary` | High | Brief, details, notes, rationale are all LLM-refined from forward + backward passes. Trust for spec work. |
| `generated_summary` | Medium | Forward pass only. Brief and details exist; notes and rationale may be absent or thin. Verify from source for formula-critical claims. |
| `extracted_summary` | Low | Only pre-existing Doxygen comments (or none). Don't rely on brief/details; go to source. |

Restoring `doc_state` as a filtered field (not exposed in EntitySummary, but present
in EntityDetail and returned by `explain_interface`) would let spec agents and auditors
quickly identify which entities need source-level verification vs. which can be read
from the MCP responses directly.

The build-time cost is zero — the state is already in the generated_docs files and
gets discarded during the current DB build. It just needs to be carried through.

## 8. Summary of Proposed Build-Time Additions

| Artifact | Source Fields | Derived Output | Primary Consumer |
|----------|--------------|----------------|------------------|
| `capability_edge_cases` table/column | `notes` from all capability members | Pre-aggregated edge case list per capability | Spec agent (`get_chunk_evidence`), Auditor |
| `is_contract_seed` flag | `rationale`, `fan_in` threshold | Flag on high-fan-in entities with rich rationale | Planning agent, contract scaffolding tools |
| `entity_usages` table | `usages` dict (exploded) | Queryable caller→callee semantic relations | Integration surface tools, `explain_interface`, auditor cross-reference |
| `notes_length` metric | `notes` | Complexity signal alongside fan_in/fan_out | `estimate_complexity`, planning agent |
| `doc_state` restoration | `state` | Trust/grounding signal on EntityDetail | Auditor, spec agent calibration |
| `spell_rationale_embedding` | `rationale` for spell_*/spec_* | Bootstrap spell cluster assignments | `get_spell_cluster` pre-seeding |
| `details_embedding` (optional) | `details` only | Higher-precision formula-hunting search | Formula extraction tools |

None of these require new LLM inference. They are aggregation and indexing operations
over the existing generated_docs corpus. The highest-value items are the
`entity_usages` table explosion and the `capability_edge_cases` aggregation — both are
pure build-time passes that unlock qualitatively different tool responses.

## 9. What This Does NOT Provide

To be clear about the gaps that the generated_docs fields cannot close, regardless of
how cleverly we index them:

**`act()` format strings.** The LLM-generated docs rarely quote exact format strings
— they describe what the player sees in prose ("sends a message to the room describing
the attack"). The actual format strings (`"$n slashes $N!"`) require static extraction
from source code. The Layer C2 work (message catalog) is genuinely separate.

**Data table values.** The `details` field for functions like `get_skill_level()` will
describe *how* the skill table is consulted, not *what values it contains*. Race stats,
class skill groups, spell mana costs — these require structured extraction from
`const.cc`/`tables.cc`, not reading the entity docs.

**Runtime ordering.** Nothing in the generated_docs captures the phase-locked
execution order of the game loop. That information is structural (which functions call
which at what cadence) and must come from the call graph combined with hardcoded pulse
constants, not from LLM annotations.

**Behavioral traces.** The docs describe what functions do; they don't record what a
specific execution of `do_kill` on a level-10 warrior attacking a level-8 goblin
produces. The Golden Trace library requires either legacy server instrumentation or
reconstructed scenario analysis.

The generated_docs fields are the highest-density pre-computed evidence source we
have. Exploiting them more fully through build-time materialization makes that
evidence queryable by view rather than just by full-text search.
