# V2 System Indexing — Problem Statement & Agreed Taxonomy

Companion to [DESIGN.md](design.md) (V1, locked) and [DESIGN_v2.md](DESIGN_v2.md)
(V2 specification). This document records the problem analysis, options evaluated,
and decisions made for the V2 system-level documentation taxonomy.

**Status:** Decisions finalized after team review. Implementation specified in
DESIGN_v2.md.

---

## 1. Context

### 1.1 What we're building

The MCP doc server (DESIGN.md) exposes the Legacy codebase as a searchable knowledge
store for AI assistants. V1 is entity-forward — every tool resolves, describes, and
traverses individual code entities (functions, classes, variables) using documentation,
a dependency graph, and capability group classification.

V2 adds a **system-level narrative layer**: curated prose describing how larger game
systems work, chunked for semantic search, and linked to code entities via a many-to-many
join table. The goal is to answer *"what larger system narrative helps interpret this
piece of code?"* — a question V1's per-entity docs cannot answer alone.

### 1.2 What we have today

The project has existing system documentation:

- **22 component docs** in `docs/components/*.md` (~6,500 lines total)
- **1 overview doc** in `docs/subsystems.md` (~720 lines, 23 numbered subsystems)

These were generated by an agent with limited holistic understanding of the codebase.
They are **not suitable for direct ingestion** in their current form.

### 1.3 The V2 schema we're targeting

From DESIGN.md §18.2, refined by team feedback into DESIGN_v2.md:

```sql
subsystems (id, name, parent_id, kind, layer, description, source_file, depends_on, depended_on_by)
subsystem_docs (id, subsystem_id, section_path, heading, section_kind, is_overview,
                narrative_role, grounding_status, body, evidence, embedding, ...)
entity_subsystem_links (entity_id, subsystem_id, role, link_source, confidence, notes, evidence)
subsystem_edges (source_id, target_id, relationship)
```

Key schema properties:
- **`parent_id`** on `subsystems` allows nesting (system → feature)
- **`kind`** on `subsystems` — closed enum: `system`, `feature`, `support` (reserved: `aspect`)
- **`layer`** on `subsystems` — closed enum: `infrastructure`, `data_model`, `game_mechanic`, `content_system`, `player_feature`, `operations`
- **`entity_subsystem_links`** is many-to-many with a `role` enum (`core`, `entry_point`, `supporting`, `utility`, `integration`)
- **`subsystem_edges`** captures system-level dependencies (`depends_on`, `bidirectional`)
- **`section_kind`** on docs is a closed enum for retrieval strategy (`overview`, `responsibilities`, `key_components`, `implementation`, `dependencies`, `behaviors`, `future`)
- **`grounding_status`** on docs — trust signal: `grounded`, `mixed`, `weak`, `rejected`
- **`evidence`** JSONB on docs — chunk-to-code provenance: `{entity_ids, capabilities, files, confidence_note}`

The schema is specified in DESIGN_v2.md. This document focuses on the taxonomy
decisions that shaped it.

## 2. Problems with the Current Documentation

### 2.1 Massive content duplication

The same systems are described in multiple places with conflicting scope boundaries:

| System | Described in |
|--------|-------------|
| Combat | `character_system.md` (§Combat System), `subsystems.md` #12 |
| Magic | `character_system.md` (§Magic System), `subsystems.md` #13 |
| Affects | `character_system.md` (§Affect System), `subsystems.md` #14 |
| MobProg / NPC AI | `character_system.md` (§Scripting and Behavior), `mobprog_system.md`, `subsystems.md` #15 |
| Data Tables | `character_system.md` (§Data Tables), `object_system.md` (§Data Tables), `subsystems.md` #7 |
| Shops | `economy.md` (§Shop System), `object_system.md` (§Shop System) |
| Marriage | `user_experience_enhancers.md`, `social_and_communication.md` |
| Note Boards | `in_game_editor.md` (§Note Boards), `subsystems.md` #16 |
| Logging | `utilities.md`, `logging_and_monitoring.md`, `admin_controls.md` |
| Scan/Hunt/Movement | `world_visualization.md`, `world_system.md` |

For DB ingestion, each feature must be owned by **exactly one** subsystem. Duplicate
prose creates conflicting chunks that degrade search quality and confuse agents.

### 2.2 Inconsistent granularity

- **Too large:** `character_system.md` is 240 lines covering ~6 distinct systems
  (character data, combat, magic, affects, skills/progression, NPC AI). It is
  effectively a mini-`subsystems.md`.
- **Too narrow:** `world_visualization.md` covers only scan, hunt, and worldmap
  rendering — arguably sub-features of the World System.
- **Arbitrary groupings:** `user_experience_enhancers.md` bundles aliases, marriage,
  and paintball — three unrelated features — into one doc by loose theme.

### 2.3 Subsystems.md vs components/ divergence

`subsystems.md` defines 23 numbered subsystems. The 22 component files don't map 1:1:

- `subsystems.md` separates Character Data Model (#8) from Character Progression (#9),
  but `character_system.md` contains both plus Combat, Magic, Affects, and MobProg.
- `subsystems.md` has Combat (#12), Magic (#13), Affect (#14) standalone, but they're
  all subsections of `character_system.md`.
- `world_visualization.md`, `status_and_look_commands.md`, `builder_commands.md`, and
  `logging_and_monitoring.md` have no corresponding numbered subsystem.

### 2.4 Inflated/hallucinated content

Several docs describe capabilities that don't exist in the codebase:

- **`event_dispatcher.md`** describes wildcard subscriptions, priority starvation
  prevention, thread safety, event flow visualization. The actual implementation is
  a single file (`Dispatcher.cc`) with a simple handler map.
- **`memory_and_gc.md`** describes atomic operations, thread contention minimization,
  cycle detection, cache-line optimization. This is a single-threaded MUD;
  `memory.cc` is 17 LOC.
- **`logging_and_monitoring.md`** describes remote logging, anomaly detection,
  capacity planning, alert escalation. Actual logging is `bug()`, `log_string()`,
  and file writes.

Hallucinated content would produce doc chunks that mislead agents about what the
code actually does.

### 2.5 Not ready for relational ingestion

For the V2 schema, we need:
- **Clean entity boundaries** — each system/feature is a distinct row
- **Clear parent-child hierarchy** — `parent_id` relationships are unambiguous
- **No duplication** — each feature described in exactly one `subsystem_docs` chunk
- **Consistent depth** — comparable granularity across systems
- **Grounded content** — prose describes what the code does, not speculation

## 3. The Classification Question

The V2 schema gives us `parent_id` for nesting. The design question is: how do we
use it? What axes of classification should drive the taxonomy?

### 3.1 Constraints from the schema and V1

1. **Capabilities ≠ systems.** The `capability` column (30 groups, typed as domain /
   policy / projection / infrastructure / utility) is already populated on every entity
   in V1. It is an orthogonal axis. A system like "Combat" spans capabilities (combat,
   affects, attributes, output, persistence). A capability like `string_ops` is a
   utility used across all systems. We don't replace capabilities with systems — we
   add a complementary layer.

2. **Entity↔system is many-to-many.** `damage()` can be `core` to Combat and
   `supporting` for Magic. `act()` can be `utility` to everything that uses output
   formatting. The join table handles this. The taxonomy doesn't need to be a
   partition — systems can share entities.

3. **Docs must chunk cleanly.** Each `subsystem_docs` row is a section of prose with
   a heading, `section_kind`, and embedding. The taxonomy determines what these
   chunks describe and how they're scoped.

4. **Agents need both navigation and search.** An agent should be able to browse
   ("list subsystems → get combat → get its features") and search ("find docs about
   spell casting interruption"). The taxonomy must support both.

### 3.2 What context-gathering actually needs

When an agent looks at `damage()` and asks for system context, the useful responses are:

1. **"This is core to the Combat System"** → needs a clear system identity
2. **"The damage pipeline works: hit check → damage calc → defensive rolls → elemental
   cascading → death processing"** → needs a narrative chunk from the Combat doc
3. **"Combat depends on Affects for status effects and Object System for equipment
   degradation"** → needs system-level edges
4. **"This function also participates in Magic (when spells deal damage)"** → needs
   many-to-many entity links with roles
5. **"The affect system applies/removes temporary modifiers with caching"** → needs
   the Affect System's own doc, reachable via the dependency edge

This tells us: **identity + navigable hierarchy + typed edges + many-to-many links**.
Not tags. Not deep nesting. Not free-text classification.

## 4. Options Evaluated

Four options were evaluated during the design process. See the git history for the
original detailed analysis. Summary of each:

### Option A: Strict Tree (System → Subsystem → Feature) — Rejected

Clean `parent_id` nesting, natural for browsing. Fails for cross-cutting systems
(Affects, Persistence) that have no natural parent. Forces arbitrary placement that
misleads agents.

### Option B: Flat Tags / Faceted Classification — Rejected

Maximum flexibility via multi-tag assignment. No navigable structure, tags drift
without governance, doesn't fit the `parent_id` hierarchy in the schema. Good as a
supplement, bad as a primary organizer.

### Option C: Shallow Hierarchy + Typed Cross-References — **Accepted**

Two levels: **System** (top-level) and **Feature** (child). Cross-cutting expressed
via edges and many-to-many entity links, not nesting. Each system carries a `layer`
and `kind` classification. Details in §5.

### Option D: Primary Hierarchy + Aspect Overlays — Deferred

Like C, but with a second orthogonal "aspects" dimension for cross-cutting patterns
(persistence patterns, messaging patterns). Elegant but adds a third classification
axis. `kind: aspect` is reserved in the schema for future use if C's edge-based
cross-references prove insufficient.

## 5. Accepted Design: Option C with Team Refinements

**Shallow hierarchy + typed cross-references**, with `kind` and `layer` columns on
the `subsystems` table, and `grounding_status` + `evidence` on doc chunks.

### 5.1 Why this shape

It maps directly to the V2 schema. The hierarchy gives identity and navigation. The
edges give cross-cutting relationships. The entity links give code-to-narrative
connections. The chunked docs with `section_kind` give targeted retrieval.

### 5.2 The `kind` enum (from team feedback)

Without `kind`, `Combat`, `Attack Resolution`, and `Utilities` are all rows in the
same table distinguished only by `parent_id` and `layer`. Adding `kind` makes the
navigational weight of each row explicit:

| Kind | Meaning | Examples |
|------|---------|----------|
| `system` | Top-level, independently navigable, has its own behavioral narrative | Combat, Magic, World System |
| `feature` | Child of a system, substantial enough for its own doc section | Attack Resolution, Spell Casting, Area Resets |
| `support` | Real topic, but lacks independent narrative — reference documentation | Utilities, Memory & GC, Event Dispatcher |
| `aspect` | *(Reserved)* Cross-cutting pattern docs | persistence_patterns, messaging_patterns |

**Qualification criteria for `kind: system`** (a top-level system should satisfy most of these):

- Has a stable user- or developer-recognizable identity
- Has its own behavioral narrative, not just helpers
- Has meaningful dependency edges to peers
- Is likely to be a navigation destination on purpose
- Can own at least one overview and a few nontrivial sections without padding

### 5.3 The `layer` enum

Six layers, with explicit boundary rules for the curation agent:

| Layer | Definition | Examples |
|-------|-----------|----------|
| **infrastructure** | Technical plumbing with no game semantics | Game Engine, Networking, Command Interpreter |
| **data_model** | Core entity structures and their accessors | Character Data, Object System, World System |
| **game_mechanic** | Rules and behaviors that define gameplay | Combat, Magic, Affects, Skills & Progression |
| **content_system** | Systems primarily about authored game content or content authoring/runtime interpretation | MobProg/Scripting, Help System, Notes & Editor |
| **player_feature** | Systems primarily about player-facing interaction loops and gameplay services | Social & Comms, Economy, Quests, Clans & PvP |
| **operations** | Admin/builder tools, persistence, operational infrastructure | Persistence, Admin & Builder Tools |

**Boundary rule for `content_system` vs `player_feature`:**
- `content_system`: the system's primary purpose is managing authored artifacts and
  their runtime interpretation (help entries, MobProg scripts, note board posts)
- `player_feature`: the system's primary purpose is a player-facing interaction loop
  or gameplay service (questing, trading, chatting, competing)

If a system straddles the boundary (e.g., Shops involve authored NPC configurations
*and* player buy/sell loops), pick the dominant purpose and link the other via edges.

### 5.4 Prose ownership rule

**Each canonical narrative chunk has exactly one owning subsystem.** Other subsystems
may reference it via cross-reference, dependency, or inclusion reasons — but the prose
lives in one place.

Example: the "damage pipeline" chunk canonically lives under Combat. Magic references
it via `subsystem_edges` (Magic → Combat) and `entity_subsystem_links` (`damage()` is
`supporting` for Magic). No duplicate damage pipeline chunk exists under Magic.

### 5.5 Schema additions (relative to DESIGN.md §18.2)

| Change | Table | Type | Description |
|--------|-------|------|-------------|
| `kind` | `subsystems` | `TEXT NOT NULL` | Closed enum: `system`, `feature`, `support` (reserved: `aspect`) |
| `layer` | `subsystems` | `TEXT` | Closed enum: `infrastructure`, `data_model`, `game_mechanic`, `content_system`, `player_feature`, `operations` |
| `grounding_status` | `subsystem_docs` | `TEXT NOT NULL DEFAULT 'mixed'` | Trust signal: `grounded`, `mixed`, `weak`, `rejected` |
| `narrative_role` | `subsystem_docs` | `TEXT` | Derived from `section_kind` + heading heuristics: `overview`, `mechanism`, `dependency`, `edge_case`, `admin`, `history` |
| `evidence` | `subsystem_docs` | `JSONB` | Chunk-to-code provenance: `{entity_ids, capabilities, files, confidence_note}` |

Full schema in DESIGN_v2.md.

## 6. Relationship to Capabilities

The four classification axes in the final design:

| Axis | Granularity | Cardinality per entity | Purpose |
|------|-------------|----------------------|---------|
| **Capability** (V1) | Function-level grouping | 1 (single assignment) | *What semantic responsibility does this code fulfill?* (combat, output, persistence, string_ops) |
| **System** (V2) | Architectural component | Many (via join table with role) | *What larger system narrative helps interpret this code?* (Combat System, Affect System) |
| **Kind** (V2) | System-level metadata | 1 (column on system) | *What sort of navigational object is this?* (system, feature, support) |
| **Layer** (V2) | System-level metadata | 1 (column on system) | *What abstraction level is this system?* (infrastructure, game_mechanic, operations) |

They are complementary:
- `damage()` has **capability** `combat`, is **core** to the **Combat** system (`kind: system`, `layer: game_mechanic`) and **supporting** for **Magic** (`kind: system`, `layer: game_mechanic`).
- `act()` has **capability** `output`, is **utility** to nearly every system, which span multiple layers.
- `save_char_obj()` has **capability** `persistence`, is **core** to the **Persistence** system (`kind: system`, `layer: operations`) and **supporting** for **Character Data** (`kind: system`, `layer: data_model`).
- `Pooled<T>` has **capability** `memory_management`, is documented under **Memory & GC** (`kind: support`, `layer: infrastructure`) — a reference entry, not a behavioral system.

No axis replaces another. Together they give agents four ways to find and filter
relevant context for any piece of code.

## 7. Key Decisions Log

Decisions made during team review, for reference:

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Option C accepted | Shallow hierarchy + edges handles cross-cutting without forced nesting |
| 2 | Add `kind` enum to `subsystems` | Prevents flat equality between systems, features, and support entries |
| 3 | Add `grounding_status` to `subsystem_docs` | Trust signal analogous to entity `doc_quality` — enables downranking of weak/hallucinated chunks |
| 4 | Add `evidence` JSONB to `subsystem_docs` | Chunk-to-code provenance improves reranking precision and curation reviewability |
| 5 | Add `narrative_role` (derived) to `subsystem_docs` | Gives agents reading order within a subsystem, derived from `section_kind` + heading heuristics |
| 6 | Keep 6 layers with explicit boundary rules | `content_system` vs `player_feature` distinction is useful but needs a one-paragraph decision rule in the curation contract |
| 7 | Chunk-first curation pipeline | Produce candidate chunks with metadata before assigning ownership — prevents premature decisions on hallucinated/duplicate content |
| 8 | "One canonical owner per chunk" | Prose duplication solved by ownership, not by restricting feature scope. Other systems reference via edges and links. |
| 9 | Canonical-conflict lint rule | Flag high-similarity chunks under different owners, repeated headings across owners, overlapping evidence sets |
| 10 | Utilities/Memory/EventDispatcher as `kind: support` | Don't have independent behavioral narratives — reference docs, not navigable systems |
| 11 | Loot Generation as `kind: feature` under Object System | Substantial code but not an independent peer-level system. Decision may be revised during curation. |
| 12 | `aspect` reserved in `kind` enum | Door open for Option D if edge-based cross-references prove insufficient |
