# Implementation Plan: Generalized Embedding Cache

**Branch**: `003-generalized-embedding-cache` | **Date**: 2026-03-20 | **Spec**: [spec.md](spec.md)

## Summary

Generalize the embedding cache persistence mechanism to support multiple embedding types (entities, usages, future types) with independent cache files. Replace the single hardcoded entity cache with a parameterized system that accepts a type identifier and generates type-specific cache files (`embed_cache_{model}_{dim}_{type}.pkl`). Entity embeddings use `_entity` suffix for consistency. Usage embeddings use `_usages` suffix. The mechanism supports unlimited future types without code changes.

## Technical Context

**Language/Version**: Python 3.14+
**Primary Dependencies**: Existing build script infrastructure (pickle, pathlib, loguru)
**Storage**: Filesystem (artifacts directory, pickle format)
**Testing**: pytest (unit tests for save/load functions)
**Target Platform**: Linux/macOS build environment
**Project Type**: Build pipeline enhancement (library function)
**Performance Goals**: Cache load/save operations complete in under 1 second
**Constraints**: No external dependencies beyond stdlib + loguru; must be called from existing build script
**Scale/Scope**: ~5,300 entity embeddings, ~2,000 usage embeddings, unbounded future types

## Implementation Order

1. **embeddings.py** — Refactor cache save/load functions to accept type identifier parameter
2. **build_mcp_db.py** — Update entity embedding cache calls to pass `"entity"` type
3. **build_mcp_db.py** — Update usage embedding cache calls to pass `"usages"` type
4. **test_embeddings.py** — Add unit tests for parameterized cache with multiple types
5. **Integration test** — Run full build and verify separate cache files created

Steps 1-3 are sequential (refactor, then update call sites). Step 4 (tests) can be developed in parallel with steps 1-3. Step 5 is the final validation.
